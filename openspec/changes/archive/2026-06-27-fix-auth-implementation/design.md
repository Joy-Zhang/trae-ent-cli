## Context

当前鉴权模块是在项目初始化时基于假设实现的，与 TRAE 企业版官方文档存在几处不一致：
1. Token 响应字段名错误（`expires_in` → `expire`）
2. 默认过期时间错误（3600s → 7200s）
3. 提前刷新时间不符合官方建议（60s → 300s）
4. 缺少对业务层 token 过期错误（code=1002）的处理

这些问题会导致真实 API 调用时鉴权失败。

## Goals / Non-Goals

**Goals:**
- 完全符合 TRAE 企业版鉴权文档规范
- 正确解析 token 响应
- 在 token 过期前主动刷新（提前 5 分钟）
- 能处理 HTTP 200 但 code=1002 的 token 过期场景，自动刷新并重试一次
- 保持现有代码结构不变，只修正有问题的部分

**Non-Goals:**
- 不改变 CLI 接口
- 不改变凭证配置方式
- 不调整 API 路径（后续变更处理）
- 不重构整个客户端架构

## Decisions

### 1. Token 响应字段解析
- **决策**：读取响应中的 `expire` 字段（不是 `expires_in`）
- **默认值**：7200 秒（2小时）
- **理由**：官方文档明确指定响应字段为 `expire`，默认 7200 秒

### 2. Token 提前刷新策略
- **决策**：在 token 过期前 300 秒（5分钟）主动刷新
- **缓存计算**：`expiresAt = now + (expire - 300) * 1000`
- **理由**：官方文档最佳实践建议提前 5 分钟刷新，避免请求失败

### 3. Token 过期重试逻辑
在 ApiClient 中增加对业务错误 code=1002 的处理：
1. 收到响应后，如果是 JSON 且 `code === 1002`
2. 清除当前内存中的 accessToken
3. 强制重新获取 token（忽略缓存）
4. 使用新 token 重试原请求 **一次**
5. 如果重试仍然失败，则正常返回错误

**与 HTTP 401 处理保持一致**：目前已有 401 重试逻辑，扩展该逻辑同时处理 code=1002。

### 4. 缓存兼容性
- **决策**：不需要做迁移兼容，旧的缓存文件会因为过期判断自然刷新
- **理由**：提前 5 分钟刷新，即使旧缓存计算有误，最多 1 小时后会自动纠正

## Risks / Trade-offs

- [无限重试风险] → 只重试一次，重试后如果仍然失败直接返回错误
- [Token 字段不匹配] → 同时兼容 `expire` 和 `expires_in` 字段，优先使用 `expire`
- [重试导致请求重复] → 仅在读接口和幂等写接口安全，非幂等接口重试可能有问题 → 本阶段先统一重试一次，后续有问题再细化
